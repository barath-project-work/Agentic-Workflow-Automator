## 1. JWT 403 Silent Failures

### Symptom
Customer service returned **403 Forbidden** with **zero logs**. Auth service's login endpoint worked fine and returned valid tokens.

### Root Cause
The `JwtAuthFilter` in customer-service caught all JWT exceptions in an **empty catch block**:

```java
// ❌ Original code — completely silent debugging
try {
    // JWT parsing...
} catch (Exception e) {
    // Nothing! No logging, no error handling
}
```

When JWT validation failed (wrong secret, expired token, etc.), the exception was swallowed silently. The security context remained empty, and Spring Security returned 403 with no explanation.

### Fix

```java
// ✅ Fixed code — warn log with context
try {
    // JWT parsing...
} catch (Exception e) {
    log.warn("JWT validation failed for {} {}: {}",
        request.getMethod(), request.getRequestURI(), e.getMessage());
    SecurityContextHolder.clearContext();
}
// Always continue the filter chain for public endpoints
filterChain.doFilter(request, response);
```

**Changes made:**
1. Added SLF4J `log.warn()` with HTTP method, URI, and error message
2. Ensured `filterChain.doFilter()` is called **after** the catch block (not inside the try)

**Lesson:** Always log exceptions in security filters. An empty catch block makes debugging impossible.

## 2. PostgreSQL `bytea` Column Type

### Symptom
Customer and opportunity list endpoints returned **500 Internal Server Error** with:
```
ERROR: function lower(bytea) does not exist
Hint: No function matches the given name and argument types.
```

### Root Cause
The `customers` and `opportunities` tables in Railway PostgreSQL were created with **`bytea` (binary) columns** instead of `varchar`/`text` for all string fields. PostgreSQL's `LOWER()` function — used by Spring Data JPA's case-insensitive search — doesn't work on binary columns.

The SQL query generated by Spring Data JPA:
```sql
SELECT ... FROM customers c1_0
WHERE lower(c1_0.name) LIKE lower(('%'||?||'%'))
-- ERROR: function lower(bytea) does not exist
```

### Diagnosis
1. Railway logs showed the full stack trace from our `GlobalExceptionHandler`
2. The stack trace included the exact SQL query being executed
3. The error clearly referenced `bytea` type and `lower()` function

### Fix (Two Approaches)

**Approach 1 — Code fix (implemented):**
```java
// CustomerService.java — fallback to findAll() when no search filters
if ((search == null || search.isBlank()) &&
    (industry == null || industry.isBlank()) &&
    (location == null || location.isBlank()) &&
    (status == null || status.isBlank()) &&
    daysSinceContact == null) {
    return customerRepository.findAll(pageable)  // Uses standard Spring Data query
            .map(this::toResponse);
}
// Only use custom JPQL query when filters are actually provided
return customerRepository.searchCustomers(...)
        .map(this::toResponse);
```

**Approach 2 — Schema fix (preferred, permanent):**
```sql
ALTER TABLE customers
  ALTER COLUMN name TYPE VARCHAR(255),
  ALTER COLUMN company TYPE VARCHAR(255),
  ALTER COLUMN email TYPE VARCHAR(255);
  -- ... repeat for all string columns
```

**Approach 3 — Entity annotation (preventive):**
```java
@Column(columnDefinition = "VARCHAR(255)")
private String name;
```

**Lesson:** Always specify `columnDefinition` on JPA entity fields. Cloud databases might create columns with unexpected types.

## 3. Railway `DATABASE_URL` Parsing

### Symptom
Spring Boot services crashed on Railway with:
```
Cannot create PoolableConnectionFactory: Connection refused: localhost:5432
```

### Root Cause
Railway provides database connection via `DATABASE_URL` environment variable in this format:
```
postgresql://user:password@host:5432/dbname
```

But Spring Boot's JDBC driver expects:
```
jdbc:postgresql://host:5432/dbname
```

The `postgresql://` vs `jdbc:postgresql://` prefix mismatch, plus the embedded credentials, cause Spring Boot's auto-configuration to fail.

### Fix

Created `DataSourceConfig.java` that checks for `DATABASE_URL`, parses it, and builds a proper JDBC DataSource:

```java
@Bean @Primary
public DataSource dataSource() {
    String databaseUrl = System.getenv("DATABASE_URL");
    if (databaseUrl == null || databaseUrl.isBlank()) {
        // Local dev fallback
        return DataSourceBuilder.create()
            .url("jdbc:postgresql://localhost:5432/atlasai")
            .username("atlasai")
            .password("atlasai_secret")
            .driverClassName("org.postgresql.Driver")
            .build();
    }
    URI uri = new URI(databaseUrl);
    String jdbcUrl = "jdbc:postgresql://" + uri.getHost() + ":" + uri.getPort() + uri.getPath();
    String[] userInfo = uri.getUserInfo().split(":");
    return DataSourceBuilder.create()
        .url(jdbcUrl)
        .username(userInfo[0])
        .password(userInfo[1])
        .driverClassName("org.postgresql.Driver")
        .build();
}
```

**Lesson:** Every Spring Boot service deployed to Railway needs a custom `DataSourceConfig.java`. Without it, the service can't connect to the database.

## 4. Redis/Kafka Dependencies on Railway

### Symptom
Services crashed at startup with Redis connection refused, even though we weren't using Redis.

### Root Cause
The `pom.xml` included:
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-redis</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.kafka</groupId>
    <artifactId>spring-kafka</artifactId>
</dependency>
```

Spring Boot's auto-configuration detects these dependencies and **tries to connect to them at startup** — `localhost:6379` (Redis) and `localhost:9092` (Kafka). Neither exists on Railway.

### Fix
Removed Redis and Kafka dependencies from production builds. Only `spring-kafka-test` remains for unit tests.

Also removed:
- `@EnableCaching` annotation
- `spring.data.redis` section from `application.yml`
- `spring.cache` section from `application.yml`
- `spring.kafka` section from `application.yml`

**Lesson:** Spring Boot auto-configures whatever it finds on the classpath. Only include dependencies you actually need at runtime.

## 5. Nginx Trailing Slash Routing

### Symptom
`GET /api/customers` returned **404** through Nginx, but worked when hitting the customer service directly.

### Root Cause
The Nginx location block was:
```nginx
location /api/customers/ {  # ❌ trailing slash
    proxy_pass ...;
}
```

A trailing slash in the location directive means it matches **only** `/api/customers/` (with trailing slash), **not** `/api/customers` (without trailing slash). The frontend's Axios calls don't include trailing slashes.

### Fix
```nginx
location /api/customers {  # ✅ no trailing slash
    proxy_pass ...;
}
```

This matches both `/api/customers` and `/api/customers/123`.

**Lesson:** Never use trailing slashes in Nginx `location` blocks for API proxying.
